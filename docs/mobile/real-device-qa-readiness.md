# REZNO Mobile Real-Device QA Runbook

## Purpose

This runbook is the post-merge gate for testing the REZNO Customer Marketplace
on a physical iPhone and Android phone. It covers repository readiness,
Development and Preview builds, installation, authenticated smoke testing, and
evidence capture. It does not authorize production publishing, App Store or
Play Store submission, production data use, or new Marketplace features.


Expected Expo result:

```text
Dependencies are up to date

```

The project uses Expo Continuous Native Generation. The absence of committed
`apps/mobile/ios` and `apps/mobile/android` directories is intentional. Native
projects generated by `expo run:*` or `expo prebuild` remain local and must not
be committed in this QA branch.

Expo prebuild can also rewrite the local `android` and `ios` npm scripts from
`expo start --*` to `expo run:*`. Record `git status --short` before and after a
local native run. Treat that rewrite as a local tool-side change and do not
silently include it in a QA commit; the canonical commands below invoke Expo
directly and do not depend on those script aliases.

## Runtime environment

The mobile API origin resolves in this order:

1. `EXPO_PUBLIC_REZNO_API_BASE_URL`
2. legacy `expo.extra.apiBaseUrl`, when an older native shell supplies it
3. `http://localhost:3000`, in a development bundle only

`localhost` on a phone is the phone itself. For physical-device QA, use the
approved HTTPS staging origin:

```text
https://rezno-staging.vercel.app
```

This value is public configuration, not a secret. Development and Preview EAS
profiles set it explicitly. Local Xcode, Android Studio, and Metro commands must
also set it explicitly as shown below.

A release bundle refuses to start without a valid HTTPS API origin. The
Production EAS environment must therefore define
`EXPO_PUBLIC_REZNO_API_BASE_URL` with the approved production origin before any
Production build is authorized; it cannot silently fall back to localhost.

Never put these server-only values in the mobile environment or QA report:

- `DATABASE_URL`
- `BETTER_AUTH_SECRET`
- session cookies
- Apple signing private keys
- Android keystore passwords
- Expo access tokens

The staging server must separately have the exact staging `BETTER_AUTH_URL`, a
staging-only `BETTER_AUTH_SECRET`, `DATABASE_URL`, and the approved admin list.

## Staging API and data gate

Run this before creating or installing a build:

```bash
API_ORIGIN=https://rezno-staging.vercel.app

curl --fail-with-body --silent --show-error \
  "$API_ORIGIN/api/auth/get-session"
curl --fail-with-body --silent --show-error \
  "$API_ORIGIN/api/commerce/public/categories"
curl --fail-with-body --silent --show-error \
  "$API_ORIGIN/api/commerce/public/stores?limit=1"
curl --fail-with-body --silent --show-error \
  "$API_ORIGIN/api/commerce/public/products?limit=1"
```

Pass criteria:

- all four requests return HTTP `200`;
- categories contain at least one visible category;
- stores contain at least one active published Store;
- products contain at least one published Product with an active Variant;
- the Variant has available inventory;
- the Store supports at least one truthful fulfillment method;
- use a Store supporting delivery for the full address and phone smoke path.

If Commerce routes return `500`, use a secure staging-only shell with the real
staging database URL. Do not paste the URL into chat, logs, screenshots, or a
tracked file.

```bash
source "$HOME/.nvm/nvm.sh"
nvm use 24

DATABASE_URL='<staging database URL supplied securely>' \
  npx --no-install prisma migrate status

DATABASE_URL='<staging database URL supplied securely>' \
  npx --no-install prisma migrate deploy

DATABASE_URL='<staging database URL supplied securely>' \
  npx --no-install prisma migrate status
```

Expected migration count at this baseline: `22`. Never use `prisma migrate
reset` against staging. After migration, repeat the HTTP gate.

If the endpoints return `200` but no Commerce Product is visible, use the
dedicated Commerce QA fixture only after the migration checks above pass. Its
guard requires PostgreSQL, an explicit `stage`/`staging` marker in the database
target, no `prod`/`production`/`live` marker, and the exact confirmation token:

```bash
DATABASE_URL='<staging database URL supplied securely>' \
COMMERCE_QA_SEED_CONFIRM=REZNO_QA_COMMERCE_STAGING_ONLY \
  npm run seed:staging:commerce-qa
```

The command creates or repairs only deterministic, namespaced QA records: an
active Store supporting delivery and pickup in Baghdad/Karrada, a published
Product, an active IQD Variant, and at least 50 available units. It is
transactional and repeatable; it does not delete records, lower `onHand`, write
`reserved`, or operate on carts, orders, or favorites. If the database target
does not carry an explicit staging marker, the command must refuse to run. Do
not weaken or bypass the guard; verify the staging database identity and its
provisioning instead.

The existing `seed:staging:marketplace-demo` command remains separate because
it seeds service discovery, not Commerce `Store`, `Product`,
`ProductVariant`, or `InventoryItem` records. Never point either QA seed at
production data. Repeat the four HTTP checks after the Commerce fixture is
ready.

## Build profiles

### Development

- includes `expo-dev-client`;
- internal distribution;
- uses the EAS `development` environment;
- embeds the staging API origin;
- requires Metro to load the current JavaScript bundle during development.

### Preview

- standalone internal build;
- uses the EAS `preview` environment;
- embeds the staging API origin;
- does not require Metro;
- is the preferred build for final realistic smoke after Development testing.

### Production

Production is deliberately outside this runbook. It uses the EAS `production`
environment and must not inherit the staging API origin.

## iPhone — local Xcode path

Prerequisites:

- macOS and current Xcode;
- Apple account added to Xcode;
- a connected, trusted iPhone;
- Developer Mode enabled on iOS 16 or newer;
- an Xcode Team selected for `com.rezno.mobile`;
- Mac and iPhone able to reach the same Metro server when using LAN mode.

Connect the unlocked iPhone by USB, trust the Mac, and verify it appears in
Xcode → Window → Devices and Simulators. Then run:

```bash
source "$HOME/.nvm/nvm.sh"
nvm use 24
cd apps/mobile

EXPO_PUBLIC_REZNO_API_BASE_URL=https://rezno-staging.vercel.app \
  npx expo run:ios --device
```

If automatic signing needs manual selection, generate the local native project
and open its workspace:

```bash
EXPO_PUBLIC_REZNO_API_BASE_URL=https://rezno-staging.vercel.app \
  npx expo prebuild --platform ios --clean

open ios/*.xcworkspace
```

In Xcode:

1. Select the REZNO application target.
2. Open Signing & Capabilities.
3. Enable Automatically manage signing.
4. Select the intended Team.
5. Select the physical iPhone as the run destination.
6. Build and run.

For subsequent JavaScript iterations:

```bash
EXPO_PUBLIC_REZNO_API_BASE_URL=https://rezno-staging.vercel.app \
  npx expo start --dev-client --lan --clear
```

Scan the QR code or select the detected bundler from the installed development
client. If LAN discovery fails, inspect the Mac firewall and confirm both
devices are on the same network before changing application code.

## Authenticated Marketplace smoke checklist

Use a staging-only email address and a phone number approved for QA. Record the
commit, build ID, API origin, device model, and OS version before starting.

### 1. Sign up and sign in

- [ ] Launch without a red screen or native crash.
- [ ] Create a fresh customer account with name, email, valid phone, and a
      password of at least eight characters.
- [ ] Confirm customer setup finishes without a retry loop.
- [ ] Sign out once, then sign in using the same credentials.
- [ ] Confirm the authenticated account state is restored.
- [ ] Confirm no session cookie, password, or full phone is printed in logs.

### 2. Market and Product

- [ ] Open the Market tab.
- [ ] Categories, Store cards, and Product cards load from staging.
- [ ] Open a Store and a published in-stock Product.
- [ ] Verify price, currency, Variant, stock state, image fallback, and back
      navigation.
- [ ] Toggle Product favorite on.

### 3. Cart

- [ ] Add the selected Variant with quantity one.
- [ ] Open Cart and verify Store, Product, Variant, price and quantity.
- [ ] Change quantity once and verify the server-authoritative version updates.
- [ ] Confirm retry/error UI does not replace a failed request with a false
      empty Cart.

### 4. Checkout

- [ ] Select delivery for the primary smoke path.
- [ ] Create a QA address with recipient, valid phone, city, area, street and
      additional details matching the Store delivery area.
- [ ] Select the address and verify it survives navigation back to Checkout.
- [ ] Review totals and offline payment method.
- [ ] Submit once; do not double-tap or manually replay with a new idempotency
      key.
- [ ] Confirm a real pending Order receipt appears.
- [ ] If testing pickup separately, confirm the customer profile has a phone
      and the Store supports pickup before submitting.

### 5. Order

- [ ] Open the created Order from the receipt.
- [ ] Verify number, Store snapshot, item snapshot, totals, fulfillment,
      payment status and history.
- [ ] Open My Activity → Orders and find the same Order.
- [ ] Do not cancel the primary evidence Order unless cancellation is a
      separately approved case.

### 6. Favorites

- [ ] Open Favorites.
- [ ] Verify the favorited Product appears.
- [ ] Favorite the Store and verify Store/Product lists remain separate.
- [ ] Toggle one favorite off and on to verify rollback and retry behavior.

### 7. Notifications

- [ ] Open Notifications after Checkout.
- [ ] Verify the Order-created notification appears.
- [ ] Load more if pagination is offered.
- [ ] Open the Order from its notification and return successfully.

### 8. Logout and session recovery

- [ ] Log out from Account.
- [ ] Confirm authenticated Commerce screens show the correct sign-in gate and
      do not leak the previous account's private data.
- [ ] Log in again.
- [ ] Confirm Order and Favorites persist from the server.
- [ ] Confirm Cart state is truthful after the completed Checkout.
- [ ] On iOS, do not treat uninstall/reinstall as a clean-session proof because
      Keychain-backed SecureStore data can persist; use explicit logout.

## Pass and stop rules

The real-device gate passes only when:

- both physical platforms install and open an approved build;
- staging pre-flight is green;
- the authenticated flow completes against real staging APIs;
- no crash, red screen, false success, false empty state, auth leak, or blocked
  primary action occurs;
- evidence is complete and sanitized.

Stop and record `FAIL` when:

- catalog endpoints are not healthy or have no approved product data;
- the build targets `localhost` or an unknown origin;
- signing, provisioning, EAS access, or Android tooling is unavailable;
- account setup cannot persist the phone needed by Checkout;
- a secret appears in logs or screenshots;
- a test would require production data, real payment, destructive migration, or
  unapproved data seeding.
